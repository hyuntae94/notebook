# gitup

작업 변경사항을 **add → commit → push** 순서로 한 번에 올린다.
커밋 메시지와 브랜치명은 사용자에게 직접 입력받는다.

## 입력 (선택)

`$ARGUMENTS`로 미리 넘길 수 있다. 형식: `"<커밋 메시지>" <브랜치명>`.
- (없음) → 아래 절차대로 사용자에게 물어본다.
- `"버그 수정" hotfix/login` → 메시지·브랜치를 바로 사용(물어보지 않음).

## 절차

1. **변경사항 스테이징 미리보기** — 먼저 무엇이 올라갈지 보여준다.

   ```bash
   git add -A && git status --short
   ```

   - git 저장소가 아니면 그 사실을 알리고 중단한다.
   - 스테이징된 변경이 하나도 없으면 "올릴 변경사항이 없다"고 알리고 중단한다.

2. **커밋 메시지 입력** — `$ARGUMENTS`에 메시지가 없으면 사용자에게
   "커밋 메시지를 입력해 주세요"라고 묻고 답변을 기다린다.

3. **브랜치명 입력** — `$ARGUMENTS`에 브랜치가 없으면 사용자에게
   "푸쉬할 브랜치명을 입력해 주세요"라고 묻고 답변을 기다린다.
   - 현재 브랜치를 알려주면 참고에 도움이 된다(`git branch --show-current`).

4. **최종 확인** — 올릴 내용을 한 줄로 요약해서 보여준다.
   예: `"<메시지>" → 브랜치 <브랜치명> 로 push 합니다.`

5. **실행** — 헬퍼 스크립트로 add → commit → push 를 수행한다.
   메시지와 브랜치명은 반드시 따옴표로 감싸 그대로 전달한다.

   ```bash
   bash .claude/skills/gitup/scripts/gitup.sh "<커밋 메시지>" "<브랜치명>"
   ```

   - 스크립트는 존재하는 브랜치면 전환, 없으면 새로 만든 뒤 push 한다.
   - 첫 줄이 `ERROR:`면 그 사유를 사용자에게 알리고 중단한다.

6. **보고** — push된 브랜치명과 커밋 해시를 요약한다.
   push 실패(인증·원격 없음 등) 시 에러 메시지를 그대로 전달한다.

## 주의

- 이 스킬은 **원격에 push** 하므로 되돌리기 어렵다. 4단계 요약을 항상 보여주고,
  사용자가 메시지/브랜치를 명시했으면 그것이 곧 진행 동의로 본다.
- `git add -A`는 모든 변경(신규/수정/삭제)을 담는다. 일부만 올리고 싶다는
  요청이면 이 스킬 대신 직접 staging 후 커밋하도록 안내한다.
