# changelog

git 커밋 히스토리를 읽어 **일자별로 정리된 `CHANGELOG.md`**를 자동 생성/갱신한다.

## 입력 (선택)

`$ARGUMENTS`로 git log 범위를 좁힐 수 있다. 없으면 전체 히스토리.
- (없음) → 전체 커밋
- `--since=2026-06-01` → 해당 날짜 이후
- `v1.0..HEAD` → 특정 범위

## 절차

1. **데이터 수집** — 스킬 디렉터리의 헬퍼를 실행한다(인자는 그대로 전달):

   ```bash
   bash .claude/skills/changelog/scripts/gather.sh $ARGUMENTS
   ```

   출력은 `branch`, `commits`(총 개수), 그리고 `---log---` 아래에
   `날짜<TAB>짧은해시<TAB>제목<TAB>작성자` 형식의 커밋 목록(최신순)이다.
   - 첫 줄이 `ERROR:`면 그 사유(예: git 저장소 아님)를 사용자에게 알리고 중단.
   - 커밋이 0개면 빈 템플릿만 만들고 그 사실을 알린다.

2. **기존 파일 확인** — 루트 `CHANGELOG.md`가 있으면 Read로 읽는다.
   없으면 새로 만든다.

3. **일자별 그룹화** — 커밋을 `YYYY-MM-DD`로 묶는다. **최신 날짜가 맨 위**.

4. **날짜 안에서 카테고리 분류** — 커밋 제목을 보고 아래로 나눈다.
   비어 있는 카테고리는 출력하지 않는다.
   - **Added** — 새 기능/파일/엔드포인트 (`feat`, add, 신규)
   - **Changed** — 동작/구조 변경, 리팩터링 (`refactor`, change, update)
   - **Fixed** — 버그 수정 (`fix`, bugfix)
   - **Removed** — 삭제/제거 (remove, delete)
   - **Docs** — 문서 변경 (`docs`, 문서)
   - **Other** — 위에 해당하지 않는 나머지

5. **항목 형식** — `- <사람이 읽기 좋은 한국어 설명> (`<짧은해시>`)`.
   `feat:`, `fix:` 같은 접두어는 떼고 자연스럽게 다듬는다.

6. **멱등 갱신(중요)** — 기존 `CHANGELOG.md`를 통째로 덮어쓰지 않는다.
   - 이미 기록된 짧은 해시는 **다시 추가하지 않는다**.
   - 새 커밋만 해당 날짜 섹션에 추가한다. 날짜 섹션이 없으면 올바른 위치
     (최신순)에 새로 만든다.
   - 사용자가 손으로 적은 설명/문장은 보존한다.

7. **보고** — 끝나면 추가/갱신된 날짜와 새로 반영한 커밋 수를 요약한다.

## 출력 형식 (예시)

```markdown
# Changelog

이 프로젝트의 주요 변경 사항을 일자별로 기록한다.

## 2026-06-06

### Added
- items CRUD 엔드포인트 추가 (`433d265`)

### Docs
- PLAN_SPEC.md 계획 명세서 작성 (`446596e`)
```
